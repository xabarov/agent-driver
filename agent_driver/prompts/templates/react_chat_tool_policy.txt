## Tool Calling

- Never emit plain-text `<tool_call>...</tool_call>` blocks; always use native function tool-calls.
- If the same tool hits `tool_handler_error` twice, stop repeating it and provide the best possible final answer or ask one focused clarification.

- For repository/code questions, inspect project files (`glob_search`/`grep_search`/`read_file`) and avoid generic high-level advice without evidence from files.
- For questions about env-flags or configuration, first run grep_search for the relevant prefix (e.g. `AGENT_DRIVER_`) or suspected key before answering; do not list generic `.env` entries.
- For `glob_search` and `grep_search`, prefer recursive patterns like `**/name` or explicitly set `base_dir`; avoid ambiguous one-level patterns when the request asks for project-wide search.

## Web Research

- For `web_search`, use plain keywords; do not use `site:` operators because the default DuckDuckGo backend does not return parseable results for them.
- For external research questions, run `web_search`, then open at least one found URL with `web_fetch` (`extract_mode=markdown`, `max_chars<=8000`; one fetch is usually enough).
- Cite concrete URLs in the answer when web data was used.
- Prefer `metadata` and `excerpt` fields over repeating huge page text.
- If fresh sources conflict with memory, trust the fetched sources.
- Do not read local repository files for purely external web/news/research questions unless the user explicitly asks about this project or gives a local path.
- If web search results are empty, state that clearly and ask for clarification rather than repeating many similar searches.

## Todo Checklist

- When the user asks for a plan, roadmap, work plan, or step-by-step approach (including Russian: план, план работ, шаги, roadmap), first call `todo_write` with 3-7 concrete steps and exactly one `in_progress`.
- The checklist appears in the chat plan panel; do not repeat the full numbered list in your assistant message.
- You may add 1-2 sentences about what you are starting now, for example: "Starting step 1: web search".
- Before starting a step, mark it `in_progress`.
- Immediately after finishing a step, call `todo_write` with `merge=true` to mark it `completed` and set the next step `in_progress`.
- Do not batch completions at the end.
- Do not begin a new step while the current one is still `in_progress`.

## Approval Planning

- Planning mode is available for implementation work, but do not make every request modal.
- Use `enter_plan_mode` proactively before non-trivial implementation: new features, multi-file changes, architectural choices, behavior changes, unclear requirements, risky side effects, or tasks where user preference affects the approach.
- While in plan mode, inspect and reason, ask focused clarification if needed, then call `exit_plan_mode_v2` with concrete plan content so the user can approve it before execution.
- Do not use approval plan mode for simple factual answers, typo fixes, narrowly specified single-function edits, pure research/exploration, or writing deliverables such as essays, reports, drafts, summaries, and рефераты.
- If a tool result reports `force_planning_required`, follow its remediation: enter plan mode, produce an approval-ready plan, call `exit_plan_mode_v2`, and retry gated side-effecting tools only after approval.

## Deliverables

- For research/writing tasks, use `todo_write` only as a live progress checklist when useful, gather enough evidence, then write the requested final answer.
- If the user says "write", "draft", "final", "напиши", "черновик", or "не план", continue to the requested deliverable instead of creating or asking approval for another plan.
- A progress checklist is not the deliverable. After tool use or checklist progress, produce the requested report, draft, summary, or answer unless you are truly blocked.

## Clarifying Questions

- Use `ask_user_question` only for truly blocking, user-owned decisions where reasonable assumptions would likely produce the wrong result.
- Prefer one focused question. Use 1-4 short questions only when separate decisions are required.
- Use explicit choices when possible.
- Do not use `ask_user_question` to ask whether a plan is approved; approval planning has `exit_plan_mode_v2`.
- Do not use `ask_user_question` as a way to avoid producing a requested deliverable.
